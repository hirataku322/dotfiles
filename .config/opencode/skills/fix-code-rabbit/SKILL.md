# GitLab CodeRabbit指摘対応スキル

GitLabのMerge Request画面を起点に、CodeRabbitの未解決指摘を一覧化し、修正・検証・返信まで一気通貫で進める。

## 目的

- 指摘を漏れなく把握する
- 影響範囲と優先度を整理して修正する
- 変更後に最低限の検証を実施する
- GitLabのディスカッションを解決状態にする

## 対応フロー

1. **MRを開く**
   - ユーザー提示のGitLab MR URLを開く
   - URLがない場合は、現在ブランチに対応するMRをGitLab上で特定する
   - 対象タブは `Overview` と `Changes` を中心に確認する

2. **GitLab上でCodeRabbit指摘を収集する**
   - `Overview` のDiscussionから、CodeRabbitの未解決スレッド（Unresolved）を優先収集
   - 必要に応じて `Changes` タブの行コメントも確認
   - 取得時は以下を最低限メモする
     - ファイルパス
     - 行番号
     - 指摘内容（要約）
     - 重要度（高/中/低）
     - スレッドURL（追跡用）

3. **重複・誤検知を仕分ける**
   - 同一原因の重複指摘は1件にまとめる
   - 仕様上問題ないものは、GitLabスレッドに「非対応理由」を明記して残す

4. **修正方針を決める**
   - 影響が小さく効果が高いものから着手
   - 大きな設計変更が必要な場合は、先に方針を明示する

5. **ローカルで修正する**
   - 既存規約・既存実装に合わせる
   - 関連箇所の副作用（型、例外、null、境界値）を確認する

6. **検証する**
   - 変更箇所に近いテスト/lint/型チェックを優先実行
   - 実行できない検証がある場合は理由と代替確認を明記

7. **GitLabで返信・Resolveする**
   - 各指摘スレッドに対応内容を返信する
   - 解消済みは `Resolve thread` を実行する
   - 非対応は理由を記載し、必要ならレビュアー判断待ちとして残す

8. **最終報告をMRに残す**
   - 「対応済み / 非対応（理由）」を整理したまとめコメントを投稿する
   - 未解決スレッド件数を明記する

## 指摘の優先度基準

- **高**: バグ、セキュリティ、データ破壊、クラッシュ、明確な仕様逸脱
- **中**: 保守性低下、可読性低下、性能劣化の可能性
- **低**: 表記ゆれ、軽微なリファクタ、好みの差分

## 実務ルール

- まずGitLab上の未解決スレッドを0件に近づける
- 1指摘につき1コミットにこだわらず、原因単位でまとめてよい
- 指摘を直す過程で見つけた関連不具合は、スコープ逸脱しない範囲で同時修正可
- 既存変更（他者作業）を破壊しない

## GitLab確認ポイント

- `Overview`: 未解決ディスカッション数、CodeRabbitの全体コメント
- `Changes`: 行単位コメントの反映状況
- `Pipelines`: 修正後のCI結果
- `Commits`: 対応コミットの差分粒度

## 報告テンプレート

```markdown
## CodeRabbit対応結果（GitLab MR）

### 対応済み

- [x] `path/to/file.ts:123`
  - 指摘: 〜〜
  - 対応: 〜〜
  - Thread: `https://gitlab...`

### 非対応（理由あり）

- [ ] `path/to/file.ts:45`
  - 指摘: 〜〜
  - 理由: 仕様上問題ないため（理由詳細）
  - Thread: `https://gitlab...`

### 検証

- 実行: `...`
- 結果: 全件成功 / 一部未実施（理由: ...）

### スレッド状況

- Resolved: X件
- Unresolved: Y件
```

## 開始時の確認事項

- GitLab MR URL
- 対象ブランチ
- 修正対象の範囲（全指摘 / 未解決のみ / 高優先度のみ）
- 返信方針（各スレッド返信 + Resolve / まとめコメントのみ）

## 注意点

- 指摘対応そのものが目的化しないよう、仕様整合性を最優先する
- AIレビューの提案は採用/不採用を必ず根拠付きで判断する
- GitLab上で追跡しやすいよう、返信にファイルパス・理由・検証結果を必ず含める
